# D2 — Split-Then-Corrupt Protocol (Decision Record)

**Project:** When Does Data Quality Predict Model Failure? A Benchmark for Knowing Before It Breaks
**Status:** Decision record — not a frozen sign-off artifact like D1. Documents *why* the protocol is structured this way, for defense in the methodology chapter and for future reference if challenged.
**Depends on:** D1 (Corruption Mechanism Specification, v1.0, frozen)
**Feeds:** D3 (Corruption Engine), D4 (Severity Grid), D10 (Quality and Drift Metric Taxonomy), D11 (Prediction Target and Evaluation Specification)

---

## Decision 1: Should data be split before or after corruption?

**Question:** Should train/validation/test splits (or LODO fold assignments) be created before corruption is applied, or should corruption be applied to the full dataset first and then split afterward?

**Decision:** Split first, then corrupt.

**Reasoning:** Creating the splits first ensures that corruption cannot influence which observations or datasets end up in train versus test. In LODO, the held-out dataset for a fold is determined before any corruption is generated, preventing leakage from corruption design into fold construction.

---

## Decision 2: Should corruption be applied to both training and test data, or only to the test side?

**Question:** Should the model be trained and evaluated on corrupted data, or should training remain clean while corruption is applied only to the held-out/test data?

**Decision:** Keep training data clean and apply corruption only to the test/held-out side. Train remains clean in the primary design.

**Reasoning:** The research question is "does data quality/drift predict Δ-performance of a model trained clean, when it meets degraded data at deployment?" Corrupting train too would conflate that question with a different one — robustness gained through training-time exposure. Keeping train clean isolates the causal quantity of interest: Δ-performance as a function of test-side corruption severity alone. Severity sweeps (D4) therefore apply only to the test side; severity does not vary by split since only one side is corrupted.

The experimental workflow is:

1. Train the model using clean training data.
2. Evaluate the model on clean test data to establish a baseline performance.
3. Apply corruption to the test data only.
4. Evaluate the same trained model on the corrupted test data.
5. Compute the performance degradation:

$$\Delta\text{-performance} = \text{Performance}_{clean} - \text{Performance}_{corrupted}$$

Corrupting training data would introduce a different research question — whether exposure to corruption during training improves robustness. By keeping training clean, the experiment isolates the effect of test-side degradation on model performance.

**Exception (out of scope for now):** training-time corruption (robustness training) is a legitimate Phase 4 extension, not part of the core design — flagged as future work, not built into D3.

---

## Decision 3: Should corruption be generated once globally or separately for each fold?

**Question:** Should a single corrupted version of the corpus be created and reused across all folds, or should corruption be generated independently within each fold?

**Decision:** Generate corruption independently for every fold, on a `(dataset, fold_id, mechanism, alpha, seed)` basis. No single global corrupted copy of the corpus is cached and reused across folds.

**Reasoning:** A globally corrupted corpus could implicitly use information from datasets that later become held-out test datasets. Independent fold-level corruption ensures that corruption generation depends only on information available within that fold and prevents corpus-level leakage. If each dataset were corrupted once and folds drawn from that fixed corrupted version, any implicit calibration choice (e.g. "8% MCAR looks realistic for dataset X") could have been informed by seeing the whole corpus rather than just that fold's train side — the same corpus-level leakage already ruled out in Phase 0 (source over-concentration).

---

## Decision 4: Which data determines the corruption parameters?

**Decision:** MCAR's $\pi_j(\alpha)$, and any MAR/MNAR functional-form parameters, are estimated using only the current fold's training data — never the held-out dataset, never the full 17-dataset corpus.

**Reasoning:** Estimating from the held-out or full corpus is a direct leakage vector: any quality metric later computed from those same statistics would be implicitly informed by the data it's meant to be evaluated against. For MAR/MNAR specifically, the functional form (conditioning feature, link function) is fixed and pre-registered ahead of time (tied to D4's severity grid) rather than adaptively fit per dataset — adaptive fitting would reintroduce the same leakage one layer deeper.

---

## Decision 5: How are quality/drift metrics paired with performance measurements?

**Decision:** Quality/drift metrics (D10) are computed after corruption injection, using the same fold and the same corrupted draw as the performance number they're paired with — metrics and performance values originating from different corruption draws are never paired, even at matched severity.

**Reasoning:** Metrics should be computed after corruption has been injected and paired directly with the corresponding performance measurement from that same corrupted realization.

**Correct pairing:**
```
Corruption Draw A
    ├── Quality/Drift Metrics A
    └── Δ-Performance A
```

**Incorrect pairing:**
```
Quality/Drift Metrics A
paired with
Δ-Performance B
```
— even if Draw A and Draw B share the same corruption severity level.

---

## Summary

| Question | Decision |
|---|---|
| Split before or after corruption? | Split first, then corrupt. |
| Corrupt train and test or only test? | Corrupt only the test side; training remains clean. |
| Global corruption or per-fold corruption? | Generate corruption independently for each fold. |
| Which data determines corruption parameters? | Training-fold data only. |
| How are metrics paired with performance? | Metrics and performance must come from the same corruption instance. |
