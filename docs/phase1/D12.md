# D12 — Root-Cause / XAI Attribution Specification

**Project:** When Does Data Quality Predict Model Failure? A Benchmark for Knowing Before It Breaks
**Status:** Draft v1.0 — design-gate document, pending David's sign-off
**Depends on:** D10 (20-feature meta-model input vector), D11 (Relative ΔM target, LODO protocol, pooled meta-model, model-family metadata)
**Feeds:** Phase 2 root-cause analysis pipeline, D7 (Phase-2-Ready Manifest schema), final thesis discussion/results chapters

---

# 1. Purpose

D11 established _that_ the meta-model predicts Relative ΔM and _how_ its predictive accuracy is scored. D12 answers a different question — now elevated to the project's **primary scientific goal**: which quality/drift metrics actually drive predicted degradation, and does this differ systematically by model family?

D12 exists because "SHAP attribution" as a one-line roadmap item is not implementable without deciding, in advance: which attribution method(s), attributed over which representation, aggregated across folds how, sliced by model family how, and — critically — what would count as evidence _against_ the project's own hypothesis. Deciding these mid-implementation risks exactly the kind of post-hoc storytelling a reviewer will flag in an interpretability section.

---

# 2. Scope

D12 specifies: attribution method, unit of attribution, cross-fold stability criteria, per-model-family stratification strategy, corroborating (non-SHAP) methods, falsifiability conditions for H1, and reproducibility requirements.

D12 does **not** specify: provenance-weighted features or conformal prediction (both D13), meta-model architecture or hyperparameters (D11), or the underlying quality metrics themselves (D10).

---

# 3. Attribution Method

**Decision:** Use **model-specific fast SHAP explainers**, not a single uniform explainer across all four families, with **permutation importance** as a mandatory corroborating cross-check.

| Model family        | Explainer                   | Rationale                                                                          |
| ------------------- | --------------------------- | ---------------------------------------------------------------------------------- |
| XGBoost             | TreeSHAP                    | Exact, fast, standard for tree ensembles                                           |
| Random Forest       | TreeSHAP                    | Same                                                                               |
| Logistic Regression | LinearSHAP                  | Exact for linear models, near-instant                                              |
| MLP                 | KernelSHAP (model-agnostic) | Only model-agnostic option available without a differentiable-explainer dependency |

**Why not one uniform explainer for all four (e.g., KernelSHAP everywhere)?** Considered and rejected. KernelSHAP is roughly two orders of magnitude slower than TreeSHAP/LinearSHAP; running it uniformly at the 47,600-row experiment scale is not tractable without a compute-budget renegotiation — the same reasoning D10 already used to exclude MMD. Using the fast exact explainers where they exist and reserving KernelSHAP only for MLP is the implementable choice.

**Known trade-off, stated explicitly rather than glossed over:** TreeSHAP, LinearSHAP, and KernelSHAP are not numerically comparable in raw magnitude — only in **within-family rank ordering**. D12 therefore never compares raw SHAP magnitudes _across_ families (§6); all cross-family comparison happens at the level of feature _rank_, not value.

**Corroborating method:** permutation importance (model-agnostic, identical across all four families) is computed alongside SHAP for every model. A finding is only reported as a root-cause driver if it agrees in direction and approximate rank between SHAP and permutation importance — this is the project's guard against a SHAP-only artifact reading as a scientific finding.

---

# 4. Unit of Attribution

**Decision:** Attribution is computed over **D10's 20-column aggregated feature vector** (mean/max of the 8 feature-level metrics, plus the 4 intrinsic dataset-level metrics), not over pre-aggregation per-original-feature values.

**Rationale:** The meta-model is trained on the 20-column vector (D11); attributing at any finer grain would require a second, un-designed model or a post-hoc disaggregation step that isn't specified anywhere and would silently expand D12's scope back into D10 territory. Attributing over what the model actually sees is the implementable, internally-consistent choice, and it directly answers the stated goal ("which _metrics_ are most predictive") without over-promising feature-level (e.g., "which original column") granularity that D10's aggregation already discarded.

**Consequence, noted for the discussion chapter:** root-cause findings will be phrased as "PSI (max)" or "Mutual Information Change (mean)" being the driver — not "column `X14` in dataset `Y`." This is a scope boundary, not a limitation to apologize for.

---

# 5. Stability Criteria

A single SHAP run on one fold is not a finding — it's a data point. D12 requires aggregation across LODO's 17 folds before anything is reported as a root-cause driver.

**Decision:**

1. Compute mean absolute SHAP value per feature, per held-out fold (17 values per feature per model family).
2. A feature is a **stable driver** for a given model family if it ranks in the **top 5 by mean |SHAP|** in **at least 12 of the 17 folds (≈70%)**.
3. Features ranking in the top 5 in fewer than 12 folds are reported as **fold-sensitive**, not drivers — explicitly distinguished in results, not silently dropped.

**Rationale for the 70% threshold:** Arbitrary thresholds invite exactly the criticism this project's own audit discipline is built to avoid (cf. D4's rejection of the "20-row absolute floor" as indefensible). 12/17 is chosen because it is stricter than a simple majority (>50%, i.e., ≥9/17) while still being defensible as "consistently, not just occasionally, top-ranked" — and it is stated numerically now rather than left to a threshold picked after seeing results.

---

# 6. Per-Model-Family Stratification

**Decision:** Do **not** train four separate meta-models. Instead, compute SHAP/permutation-importance attributions on the **single pooled meta-model** (per D11 Decision 4), then **group the resulting per-sample attribution values by the `model_family` metadata column** already present in the pooled model's input (per D11).

**Rationale:** Training four separate meta-models would quarter the effective LODO training data per family and reintroduce a design decision D11 already closed (Decision 4: one pooled model). Post-hoc grouping of a pooled model's attributions by family is the standard, implementable way to ask "does feature importance differ by family" without reopening D11. This does mean attributions are conditional on one shared pooled model rather than four independently-optimized ones — noted as a limitation, not hidden.

**Reporting format:** a 4-family × 20-feature stable-driver matrix (Y/N per §5's criterion), computed once per family using only that family's rows from the 17-fold attribution history.

---

# 7. Falsifiability — Pre-Registered Before Results Exist

Stated now, before any Phase 2 run, so root-cause claims can't be shaped after seeing the data:

**H1 (root-cause form) is supported only if:**

1. At least one Distribution or Information metric (D10 §4.2/§4.4) is a stable driver (§5) for at least two of the four model families, **and**
2. That stable driver is not simply Missing Rate or Imbalance Ratio alone (i.e., the finding isn't reducible to "more missing data is worse" — the whole point of H1 is that quality/drift signal beyond raw missingness/imbalance predicts degradation).

**H1 (root-cause form) is refuted or unsupported if:**

- No metric clears the 12/17 stability bar for any family, **or**
- The only stable drivers are Tier-0-equivalent single metrics (e.g., PSI alone dominates and nothing else is stable) — this would indicate the meta-model isn't extracting complementary signal beyond what a simple baseline already captures, directly undermining the complementarity argument D10 §2 is built on.

This falsifiability condition is the direct root-cause analogue of D11's H1a/H1b (MAE comparison) — same hypothesis, evaluated through attribution instead of prediction accuracy.

---

# 8. Reporting Artifacts

1. **Per-family stable-driver matrix** (§6) — the primary root-cause table.
2. **SHAP summary plots**, one per model family, over the pooled model's attributions restricted to that family's rows.
3. **SHAP dependence plots** for every stable driver, split by mechanism class (D1's 6 classes) — directly tests whether a metric's importance is concentrated in specific corruption regimes, which is the regime-conditional claim H1 is actually built on.
4. **Mechanism-class × severity-band interaction summary** (reusing D11's existing stratified reporting structure, §"Hypothesis Evaluation") cross-tabulated against the stable-driver matrix — this is where the "regime-conditional" part of H1 gets its evidentiary backing, not just a general "quality predicts degradation" claim.

---

# 9. Reproducibility Requirements

- SHAP/permutation-importance computation runs through one pipeline entry point per model family: `(trained_meta_model, held_out_fold_features, model_family) → attribution_vector`.
- Explainer library and version pinned (`shap` package version recorded in code, consistent with D10 §9's estimator-versioning discipline).
- Random seeds for KernelSHAP's sampling approximation (MLP only — TreeSHAP/LinearSHAP are exact, no sampling) fixed and logged per fold.
- The 17-fold attribution history (not just the final aggregated stable-driver matrix) is retained as an intermediate artifact, so the stability calculation itself is auditable and re-derivable without rerunning SHAP.

# 10. Optimization Strategy (Summary of Implementation-Easy Decisions)

| Decision point      | Choice made                                               | Why it's the easy-to-implement option                                                   |
| ------------------- | --------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| Attribution method  | TreeSHAP (XGB/RF), LinearSHAP (LR), KernelSHAP (MLP only) | Avoids uniform KernelSHAP's intractable compute cost at 47,600-row scale                |
| Corroboration       | Permutation importance, always computed alongside         | Model-agnostic, near-zero extra implementation cost, guards against SHAP-only artifacts |
| Unit of attribution | D10's existing 20-column vector                           | No new representation to build; reuses what D11's meta-model already consumes           |
| Per-family analysis | Post-hoc grouping of one pooled model's attributions      | Avoids training 4 separate meta-models / reopening D11 Decision 4                       |
| Stability rule      | Top-5 by mean \|SHAP\|, in ≥12/17 folds                   | Single fixed numeric rule, stated pre-hoc, no post-hoc threshold shopping               |
| Falsifiability      | Pre-registered in §7, before any Phase 2 run              | Prevents post-hoc narrative fitting; directly reusable in the discussion chapter        |

# 12. Future Work (Explicitly Deferred)

1. Feature-level (pre-aggregation) attribution, if D10's mean/max collapse is later judged too coarse for a specific finding worth drilling into.
2. Family-specific meta-models with independent attribution, if the pooled-model-grouping approach (§6) is found to obscure a family-specific effect too strongly.
3. Formal statistical testing of attribution differences between families (e.g., is family A's dependence on Mutual Information Change significantly different from family B's) — currently descriptive/tabular only.
