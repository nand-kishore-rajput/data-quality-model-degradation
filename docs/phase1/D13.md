# D13 — Provenance-Weighted Features and Group-Aware Conformal Prediction Specification

**Project:** When Does Data Quality Predict Model Failure? A Benchmark for Knowing Before It Breaks
**Status:** Draft v1.0 — design-gate document, pending David's sign-off
**Depends on:** D10 (20-feature vector, versioning amendment below), D11 (Relative ΔM target, LODO protocol), Phase 0 provenance decisions (HELOC sentinel story, bank-marketing `pdays`, dataset-sourcing exceptions)
**Feeds:** Phase 2 meta-model (extended input), Phase 2 evaluation (interval coverage reporting)
**Sequencing:** Formally deferred until after D12's root-cause pipeline is proven against the v1 (20-column) feature vector. This document exists so the extension is designed now and implemented later, not designed later under time pressure.

---

# 1. Purpose

This project's two genuine novelty bets — provenance-weighted quality features and group-aware conformal prediction intervals — currently had no governing design document. D13 closes that gap. It does not require Phase 2 code to change yet; it requires the _design_ to exist so that when D12's root-cause pipeline is proven, extending it doesn't require a retrofit.

---

# 2. Versioning Amendment to D10 (Prerequisite, Applies Now)

**Decision:** D10's 20-column feature vector (§6 of D10) is designated **v1**. D13's provenance features are designated **v2 additions**, appended as new columns without renumbering or removing any v1 column.

This is a one-line dated amendment to D10, not a reopening of D10's frozen taxonomy (D10 Decision 1's freeze still applies to the 12 metric _definitions_ — D13 adds a 13th category, Provenance, rather than modifying any existing one). Any pipeline, model, or attribution code built against "the current feature set" (per D12 §4) rather than a hardcoded literal `20` will accept v2 columns without modification.

# 3. Provenance Features

## 3.1 What "provenance" means operationally

**Decision:** Provenance is operationalized as a **per-dataset provenance confidence score**, not a per-cell or per-column scheme. A per-cell scheme (e.g., "this specific value's provenance confidence") would require metadata Phase 0 never collected and cannot be retrofitted without re-auditing all 17 datasets' original sources cell-by-cell — not implementable within this project's remaining timeline. A per-dataset scalar is directly implementable using information Phase 0 and D1/D4 already produced.

## 3.2 Scoring rubric

**Decision:** A single composite score `prov_j ∈ [0, 1]` per dataset `j`, computed from four equally-weighted sub-scores already implicit in this project's own documentation:

| Sub-score                 | Definition                                                                                                                                                                                        | Source                |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------- |
| `source_reliability`      | 1.0 if raw/primary source (e.g., raw ProPublica file, raw FICO file); 0.5 if a curated/derived OpenML version; 0.0 if source pedigree undocumented                                                | Phase 0 corpus notes  |
| `defect_documentation`    | 1.0 if the dataset has a documented, citable natural-defect finding (HELOC sentinels, bank-marketing `pdays`, road-safety missingness); 0.5 if profiled but no citable finding; 0.0 if unprofiled | D3/D4/D6 findings     |
| `annotation_confirmation` | 1.0 if the dataset's validity/domain constraints were manually confirmed (not just auto-suggested) per Kishore's annotation pass; 0.5 if partially reviewed; 0.0 if unreviewed                    | D1 §8 annotation file |
| `screening_pass_quality`  | 1.0 if the dataset passed Phase 0's XGBoost learnability screen without exception rationale; 0.5 if passed with a documented exception; 0.0 otherwise                                             | Phase 0 screening log |

`prov_j = mean(source_reliability, defect_documentation, annotation_confirmation, screening_pass_quality)`

**Rationale for equal weighting:** Any non-uniform weighting scheme would itself need justification this project has no basis for yet (no validation data exists on which sub-score matters more). Equal weighting is the defensible, simplest starting point — explicitly flagged in Future Work (§7) as revisable once Phase 2 results exist to justify reweighting.

## 3.3 How provenance enters the feature vector

**Decision:** Two v2 columns are added per existing feature-level metric group, not a full re-weighting of all 8 feature-level metrics individually:

1. **`prov_j` itself** — one scalar column, dataset-level, identical treatment to D10's existing intrinsic Class metrics (§4.5 of D10).
2. **Provenance-weighted Distribution/Information mean** — a single additional column: `mean(PSI, Wasserstein, JS, Entropy Shift, MI Change) × prov_j`, i.e., one composite provenance-weighted drift/information signal, not eight separately-weighted duplicates of every existing column.

**Rationale for not weighting all 8 columns individually:** Multiplying every one of the 8 feature-level metrics by `prov_j` individually would add 16 new columns (8 metrics × mean/max) that are perfectly collinear with the existing 16 (since they're just the same values scaled by a per-dataset constant) — this would harm, not help, the meta-model and directly contradicts D10 §5.4's own reasoning for excluding standard deviation ("weaker independent information ... increases dimensionality without clear scientific justification"). One composite weighted signal plus the raw `prov_j` scalar (2 new columns total) is the information-preserving, dimensionality-disciplined choice consistent with D10's existing philosophy.

**Total v2 additions: 2 columns. Final vector: 22 columns (20 v1 + 2 v2).**

# 4. Group-Aware Conformal Prediction

## 4.1 Method

**Decision:** Use **split conformal prediction via the MAPIE library**, with **Mondrian (group-conditional) conformalization grouped by dataset**, not a single pooled conformal calibration.

**Rationale for split conformal over full/jackknife conformal:** Split conformal requires only one held-out calibration set per fold and is computationally trivial to bolt onto the existing LODO structure — full conformal would require retraining the meta-model many times per fold, which is not tractable at this project's scale and timeline. This mirrors D11's own reasoning for choosing simple, LODO-compatible protocols over more elaborate alternatives.

**Rationale for grouping by dataset (Mondrian) rather than pooling all residuals:** Pooling calibration residuals across all 17 datasets risks exactly the kind of cross-dataset leakage this project's entire design (D2 Decision 3, LODO in D11) exists to prevent — a residual distribution learned partly from datasets similar to the held-out one would understate true interval width for genuinely novel datasets. Grouping calibration by dataset identity (Mondrian conformal, a standard, off-the-shelf MAPIE feature) is the direct implementation of "group-aware," matching the term already used in this project's own hypothesis language.

## 4.2 Coverage target and calibration split

**Decision:**

- Target coverage: **90%** (α = 0.10) — the standard default, chosen over a stricter 95% because with only 17 datasets and LODO's already-small per-fold sample sizes, a 95% target would require wider, less useful intervals; 90% is the standard, defensible starting point.
- Calibration set: within each LODO training split (16 datasets), hold out **20% of the pooled training rows** (stratified by dataset) purely for conformal calibration, not used for meta-model fitting in that fold. This reuses D2's existing "estimate parameters only from data not seen elsewhere" principle, applied to the conformal calibration step specifically.

## 4.3 Evaluation

**Decision:** Report **empirical coverage** (fraction of true Relative ΔM values falling inside the predicted interval) and **mean interval width**, per held-out dataset, in the same LODO-stratified table D11 already uses for MAE. A conformal method is judged successful if empirical coverage is within a pre-registered ±3 percentage points of the 90% target, averaged across the 17 LODO folds — stated numerically now, not chosen after seeing results, for the same reason D12 §7 pre-registers its falsifiability conditions.

# 5. Sequencing (Restated)

D13 is a design specification only at this stage. Implementation order:

1. D12's root-cause pipeline is built and validated against the v1 (20-column) vector.
2. D13's 2 provenance columns are added (v2), and the root-cause pipeline (already written against "current feature set," per the D10 amendment in §2) is rerun without modification to confirm it accepts the extended vector.
3. Conformal prediction is added as a downstream evaluation layer on top of the already-trained meta-model — it does not require retraining or re-validating the root-cause attribution work.

This sequencing means D13 adds no schedule risk to D12: the extension is additive by design, not a redesign.

# 6. Optimization Strategy (Summary of Implementation-Easy Decisions)

| Decision point             | Choice made                                                          | Why it's the easy-to-implement option                                                                                      |
| -------------------------- | -------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| Provenance granularity     | Per-dataset scalar, not per-cell/per-column                          | Implementable from information Phase 0 already produced; a per-cell scheme is not recoverable at this stage                |
| Provenance scoring         | 4 equally-weighted sub-scores, mean-aggregated                       | No validation data exists yet to justify unequal weights; equal weighting is the defensible default                        |
| Feature vector integration | 2 new v2 columns (raw score + one composite weighted signal), not 16 | Avoids collinearity/dimensionality blowup; consistent with D10's own std-dev exclusion reasoning                           |
| Conformal method           | Split conformal (MAPIE), not full/jackknife                          | Computationally trivial to bolt onto existing LODO folds; avoids expensive retraining                                      |
| Conformal grouping         | Mondrian, grouped by dataset                                         | Directly implements "group-aware"; prevents cross-dataset calibration leakage, consistent with D2/D11's leakage discipline |
| Coverage target            | 90% (α=0.10), ±3pp tolerance pre-registered                          | Standard default; realistic given only 17 datasets; avoids post-hoc threshold picking                                      |

# 8. Future Work (Explicitly Deferred)

1. **Reweighting the 4 provenance sub-scores** based on which one, if any, Phase 2 results show carries the most signal (currently equal-weighted by necessity, not conviction).
2. **Per-cell or per-column provenance**, if Phase 2 results suggest the per-dataset scalar is too coarse to capture within-dataset provenance variation (e.g., HELOC's own two-tiered sentinel structure per D4 §3.3 suggests provenance quality isn't even uniform _within_ one dataset).
3. **Full/jackknife conformal** as a robustness check against split conformal, if interval width proves too wide to be useful in practice.
4. **Conformalized quantile regression**, an alternative to split conformal that could produce tighter, non-symmetric intervals — deferred as a heavier dependency not justified for this version.
