# D3 — Corruption Engine: Complete Build Report

**Project:** When Does Data Quality Predict Model Failure? A Benchmark for Knowing Before It Breaks
**Scope:** Full implementation and real-corpus validation of D1's Corruption Mechanism Specification — 6 mechanism classes, 14 sub-mechanisms, 6 Python modules, 6 runner scripts.
**Status:** Complete. All six modules built, synthetic-tested, real-corpus-validated against the actual 17-dataset processed corpus, and — where real bugs were found — fixed and re-verified against real data, not just re-tested synthetically.
**Companion document:** `D3_implementation_decisions_and_findings.md` (concise reference: constants, findings, bug list). This document is the full narrative — what was built, how it was tested, and exactly what went wrong and why, in enough detail to stand alone without needing the conversation history it came from.

---

## 1. Overview

D1 (Corruption Mechanism Specification) defines the corruption engine's contract: given a dataset fold, a mechanism, a severity, and a seed, produce a corrupted version of that fold plus metadata describing what was done. D3 is the implementation of that contract. Six mechanism classes were built, matching D1's six sections:

| D1 Section               | Module                   | Sub-mechanisms                                                   | Status                     |
| ------------------------ | ------------------------ | ---------------------------------------------------------------- | -------------------------- |
| §4 Missingness           | `missingness.py`         | MCAR, MAR, MNAR                                                  | Built, real-data confirmed |
| §5 Feature Noise         | `feature_noise.py`       | Additive Gaussian, Categorical Swap, Quantization                | Built, real-data confirmed |
| §6 Label Noise           | `label_noise.py`         | Uniform, Class-Conditional (Boundary-Aware deferred per D1 §6.3) | Built, real-data confirmed |
| §7 Distribution Shift    | `distribution_shift.py`  | Covariate, Label, Slice-Conditional Concept                      | Built, real-data confirmed |
| §8 Validity Violations   | `validity_violations.py` | Constraint-based (no_negative / range / none)                    | Built, real-data confirmed |
| §9 Uniqueness Violations | `uniqueness.py`          | Global Duplication, Slice-Conditional Duplication                | Built, real-data confirmed |

**14 of 14 sub-mechanisms implemented** (boundary-aware label noise is the one D1-scoped exception, deliberately deferred by D1 §6.3 itself, not an omission).

Every module was built following the same discipline: implement against D1's mathematical specification, test against synthetic data covering edge cases, then test against the real 17-dataset processed corpus. That last step mattered more than expected — four of the six modules had a genuine bug that only surfaced on real data, invisible to any amount of synthetic testing because the bugs depended on properties real data has and hand-constructed test data didn't (pre-existing missingness, rare categories, correlated outlier-heavy features).

## 2. Architecture

### 2.1 Interface Contract

Every module exposes the same shape, agreed before any code was written:

```
corrupt(fold, mechanism, severity, seed, train_fold_reference, ...) -> (corrupted_fold, metadata)
```

- `fold` — the fold to corrupt. Per D2 Decision 2, callers are responsible for passing only the test fold; no module has an opinion on which fold it's given.
- `train_fold_reference` — the clean train fold, used only for parameter estimation (D2 Decision 4: conditioning-feature selection, z-score reference statistics, empirical marginals, bin edges, PC1 direction, IQR bounds, frozen-slice bounds). Never itself corrupted or read from for anything but reference statistics.
- `metadata` — a `CorruptionMetadata` object (defined once, in `base.py`, shared by every module): dataset name, mechanism, severity, seed, achieved rate(s), excluded columns (if any), a validation-passed flag, and a `validation_details` dict with mechanism-specific diagnostics.

### 2.2 Shared Infrastructure (`base.py`)

Started small (just the metadata schema and two math primitives for `missingness.py`) and grew as genuinely shared logic was identified across modules, rather than being duplicated:

- `CorruptionMetadata` — the shared return-value schema, including `excluded_columns` (added when per-column exclusion was introduced) and `conditioning_feature` (reused across missingness, distribution shift, and uniqueness for different purposes).
- `sigmoid`, `reference_zscore`, `solve_logit_offset_for_target_rate` — math primitives for missingness's MAR/MNAR calibration.
- `rate_tolerance`, `reference_variance_issue`, `nan_adjusted_target` — moved here from `missingness.py` once `feature_noise.py` needed the identical logic (avoiding two copies drifting apart, the same single-source-of-truth concern raised earlier for the runner scripts' `TARGET_COLUMNS` mapping).
- `select_frozen_slice_bounds` / `select_frozen_slice` — the shared, deterministic frozen-slice function, built specifically so `distribution_shift.py` (§7.3) and `uniqueness.py` (§9.2) could provably operate on the identical slice S, per D1's requirement that the two mechanisms share one frozen region. Deterministic-and-cached rather than persisted-to-disk, deliberately: a stale on-disk slice surviving a code change would silently violate "frozen from this fold," which is exactly the kind of undocumented cross-run state this project's audit philosophy exists to prevent.

## 3. Module 1: `missingness.py` (D1 §4)

**Mechanisms:** MCAR (`P(R=0) = π_j(α)`), MAR (`P(R=0) = σ(α·z(X_ik))`, conditioning feature k selected by highest train-fold correlation), MNAR (`P(R=0) = σ(α·z(X_ij))`, depends on the column's own value).

### Design notes

- D1's MAR/MNAR formula has no intercept term. Taken literally, `σ(0) = 0.5` at `z=0` regardless of severity — meaning even near-zero severity would produce ~50% missingness for typical rows. A calibration offset (solved via bisection so the achieved rate matches the target) was added to fill this gap, flagged explicitly as a D3 decision rather than a reinterpretation of D1's math.
- MNAR's mechanism-fidelity check (missingness significantly associated with the feature's own pre-corruption value) is explicitly blocking per D1 §4.3 — a draw that reduces to MCAR-equivalent behavior is invalid, not merely suboptimal.

### Testing journey and bugs found

**Round 1 — tolerance miscalibration.** The first real-corpus run (`--datasets phoneme churn`) showed 10/90 rows with `validation_passed=False`, concentrated at higher severities. Initial hypothesis (a real RNG bias at a specific call position) was tested against 500 seeds and disproven — pure coincidence from only testing 3 seeds. The actual issue: a flat `RATE_TOLERANCE = 0.02` tolerance doesn't scale with the true sampling noise, which follows `sqrt(severity·(1-severity)/n)`. At low severity the flat band was too loose; at high severity, too tight — producing a severity-dependent false-positive rate (~11% observed). Fixed with a binomial-standard-error-based tolerance (`RATE_TOLERANCE_Z = 3.5`), re-verified at ~1% false-positive rate, consistent with the tolerance's own stated confidence level.

**Round 2 — pre-existing NaN propagation.** Running against the full 17-dataset corpus, Airbnb's MAR and MNAR results showed achieved rates of exactly `0.0` for most columns at every severity — not noise, a complete failure to inject any missingness. Root cause: Airbnb has substantial genuine pre-existing missingness (a documented Phase 0 realism anchor), and when the conditioning/target column already has NaN, `reference_zscore` produces NaN in the z-array. The calibration bisection's `sigmoid(...).mean()` then returns NaN (numpy's NaN propagation), and every comparison against NaN evaluates False — so the bisection always took the same branch and collapsed to its -50 floor regardless of target severity. Fixed by calibrating against only non-NaN z-values, plus a `nan_adjusted_target` helper that scales the calibration target so the achieved rate (measured across all rows, including permanently-ineligible NaN ones) still lands on the requested severity. A related fix was needed in the association test (`scipy.stats.pointbiserialr` doesn't handle NaN internally).

**Round 3 — EEG's all-or-nothing blocking design.** Running `--all`, EEG's MNAR showed 0/15 — every draw failed. Initial hypothesis (a near-constant reference column) was tested and disproven directly against the real correlation data: AF3 showed a real, weak, borderline-significant correlation (0.005 to 0.04, occasionally crossing significance), not zero. The actual issue was structural: the blocking check required every corrupted column in a call to pass, so one persistently borderline column (AF3) invalidated four other columns (F7, F3, FC5, T7) that were working correctly every time. Redesigned to per-column granularity: the fidelity check now runs before the mask is applied, per column; a failing column is excluded from that call's output (and reported in `excluded_columns`) rather than invalidating the whole call. Raises only if every requested column fails. Re-verified against real data: EEG's MAR/MNAR calls now succeed with partial exclusions logged explicitly, rather than failing outright.

### Final real-corpus result

765/765 calls completed (750 OK, 15 PARTIAL_SUCCESS — all in EEG, expected given its documented borderline columns), 0 unexpected errors.

## 4. Module 2: `feature_noise.py` (D1 §5)

**Mechanisms:** Additive Gaussian Noise (`X' = X + eps`, `eps ~ N(0, (alpha*s_j)^2)`, applied to continuous features only), Categorical Swap (resample from the empirical marginal excluding the cell's own value), Quantization/Bin Collapse (`k = k_max - alpha*(k_max - k_min)`, equal-frequency binning).

### Design notes

- D1 restricts Gaussian noise to continuous features but doesn't specify `k_max` for quantization. Set per-column as `min(train-fold distinct value count, 20)` — flagged as a D3 decision, `QUANTIZATION_K_MAX_CAP = 20`.
- Quantization is structurally different from every other mechanism in the codebase: severity controls coarseness applied uniformly to every value, not a per-cell selection rate. No `achieved_rate`-as-proportion; the metadata instead reports achieved vs. target bin count. `seed` is accepted for interface consistency but genuinely unused (the mechanism is deterministic given the reference and k).
- Pre-existing NaN required no special handling for Gaussian noise — `NaN + noise = NaN` falls out of ordinary floating-point arithmetic.

### Testing journey and bugs found

**Chi-square bin-dropping crash.** The first real-corpus `--all` run showed 150/765 draws crashing with a scipy error: "the sum of the observed frequencies must agree with the sum of the expected frequencies." The pattern was deterministic, not intermittent — every dataset with at least one low-frequency category in its categorical marginal failed 15/15; every dataset without one (electricity, bank-marketing, ACSPublicCoverage) passed 15/15. Root cause: the marginal-preservation chi-square check dropped bins with expected count under 5 via boolean indexing before calling `scipy.stats.chisquare` — but dropping bins breaks scipy's required invariant that observed and expected totals match exactly, since the dropped observed and dropped expected counts don't generally cancel. Fixed by pooling low-expected-count categories into a single combined "other" bin instead of dropping them, which preserves both totals exactly while still satisfying the standard expected-count-of-5-or-more rule of thumb. Re-verified: reproduced the exact crash with a forced rare category (expected count around 2), confirmed the pool triggers and zero crashes result; re-ran the full corpus and all 150 previously-crashing draws passed.

### Final real-corpus result

Gaussian and Quantization were clean on their very first real run (255/255 each). Categorical Swap: initially 45 OK / 150 crashed / 4 skipped (no eligible columns); after the fix, 705 OK plus 4 skipped, 0 crashes, 0 partial successes.

## 5. Module 3: `label_noise.py` (D1 §6)

**Mechanisms:** Uniform (`P(Y'!=Y) = alpha` symmetric), Class-Conditional (asymmetric directional rates, ratio anchored to a Cleanlab confident-learning pass per D1 §6.2 — never asserted from intuition). Boundary-Aware (D1 §6.3) is explicitly deferred, not implemented — calling it raises `NotImplementedError` with the D1 rationale (probe/evaluation-family circularity risk) rather than silently doing nothing.

### Design notes

- Severity is capped below 0.5 for binary labels — at or above a 50% flip rate, Y becomes independent of X, which is total signal destruction, not "high severity." This is enforced, not just documented.
- If Cleanlab is unavailable, class-conditional noise raises an ImportError rather than silently falling back to a symmetric ratio — a symmetric fallback would quietly reduce class-conditional noise to uniform noise, defeating the mechanism and violating D1 §6.2's anchoring requirement.
- A known, flagged tension: the Cleanlab estimate uses logistic regression as its label-quality probe, and logistic regression is one of the four evaluated model families. The circularity risk is much milder here than D1 §6.3's boundary-placement concern (this only estimates a single scalar ratio), but it's real and worth revisiting when the final pipeline is assembled.

### Testing journey and bugs found

**Wrong Cleanlab API name.** The first attempt to test the real Cleanlab estimation path failed with an ImportError even though Cleanlab was installed. The guard import referenced a function that doesn't exist in the installed Cleanlab version (2.9) — the correct function is `compute_confident_joint`. Fixed by correcting the import; re-verified the estimation path works end-to-end, including a sanity check that the estimated ratio responds correctly to injected asymmetric label errors.

**Roughly fifteen-times redundant computation.** After the real-corpus run showed 510/510 OK with zero problems, a closer look revealed the Cleanlab estimate — an expensive 3-fold cross-validated logistic regression fit — was being recomputed from scratch on every one of 15 severity/seed calls per dataset, even though the estimate depends only on `train_fold_reference`, `target_col`, and `feature_cols`, none of which vary across severity or seed. Fixed with a cache keyed on object identity of `train_fold_reference` plus `target_col` and `feature_cols` — object identity rather than dataset name, since dataset name is caller-supplied and could be blank or accidentally reused, whereas object identity is correct as long as the same DataFrame object is reused across calls (which is exactly the runner's actual usage pattern). Verified directly: 15 calls with the same train fold now produce exactly 1 "fresh" computation and 14 "cached" hits, ratio provably identical across all 15; a genuinely different DataFrame object correctly triggers a fresh computation. Measured speedup: roughly 15x at smoke-test scale (0.94s versus about 14.15s per dataset); confirmed against the real corpus manifest afterward — every one of the 17 real datasets showed exactly 1 fresh plus 14 cached computations, exactly as predicted.

### Final real-corpus result

510/510 OK (17 datasets times 2 mechanisms times 5 severities times 3 seeds). 509/510 validation_passed=True; the one miss (default_credit, class-conditional, severity=0.1, seed=3) traced to a 3.51-sigma deviation against a 3.5-sigma tolerance — a boundary-case statistical miss, not a defect. One dataset (online_shoppers) showed a notably higher estimated asymmetry ratio (6.73 versus 0.6-3.7 for the rest) — flagged as worth a second look, not yet resolved.

## 6. Module 4: `distribution_shift.py` (D1 §7)

**Mechanisms:** Covariate Shift (PC1-weighted importance resampling, `P(select) proportional to exp(alpha*g(x))`), Label Shift (class-stratified resampling toward a target prior), Slice-Conditional Concept Shift (label flips restricted to the frozen slice S, features untouched).

### Design notes

- Covariate shift is implemented as weighted resampling with replacement of the fold's own rows, rather than any transformation of feature values. This is what makes D1's preservation requirements (P(Y|X) for covariate shift, P(X|Y) for label shift) hold by construction: each resampled row keeps its own original (x, y) pairing (covariate shift) or is drawn only from within its own class (label shift), so the conditional relationship literally cannot change.
- PC1 is fit on the train fold and deliberately left as the sole generator — the direction of maximum variance, not the direction most predictive of Y. This is intentional per D1 §7.1: a benchmark where drift magnitude and degradation magnitude always moved together would be trivially solvable.
- Concept shift is the one sub-mechanism confirmed to use the shared frozen slice S from `base.py`, not an independently computed one — see the cross-mechanism section below for the verification.

### Testing journey and bugs found

**Degenerate resample collapse — the most serious bug found across all six modules.** The real-corpus run showed every dataset's covariate shift PSI rising smoothly with severity, monotonically, except jm1: 0.045, then 2.300, 7.326, 7.895, 7.895 — an order of magnitude beyond every other dataset, and the exact repeated value at the two highest severities was the tell that something had saturated rather than merely drifted. Diagnosed directly, not guessed at, by instrumenting the actual weight computation: jm1's software-complexity features are correlated and outlier-heavy, and the exponential weighting formula is mathematically unbounded — when PC1 is dominated by even one or two extreme points, the exponential doesn't just up-weight them, it can put effectively 100% of the resampling probability mass on a single row. Confirmed precisely: effective sample size dropped to 1.0 out of 2000 by severity 0.15. The "corrupted" fold at that point was not a shifted distribution, it was roughly 2000 duplicates of one row. This would have silently produced meaningless downstream results for jm1's covariate-shift condition while every status check reported complete success.

Fixed by clipping the PC1 score to a bounded number of its own standard deviations (constant `G_CLIP_Z`, value 6.0) before exponentiating — preserves the mechanism's intent (stronger selection toward one tail as severity rises) while preventing single-point domination. Added an effective-sample-size diagnostic to metadata so any future instance of this is visible rather than silently passing. Re-verified against real data: jm1's PSI curve is now 0.020, 0.070, 0.246, 0.565, 1.186 — smooth and in the same range as heloc and default_credit. A milder residual was found and correctly flagged by the new diagnostic: jm1 still trips the effective-sample-size floor at severity 0.25 specifically (effective sample fraction 0.046, reproducible across all 3 seeds) — but with a reasonable PSI (about 1.2, not a collapse), this is a genuine property of jm1's data at high severity, not a remaining bug.

A related runner-script gap was found and fixed in the same pass: the diagnostic fields were computed but never written to the manifest CSV, only inferable from validation_passed being False. Fixed to write them explicitly, plus live console logging when a resample is flagged degenerate.

### Final real-corpus result

765/765 OK. All 17 datasets show smooth, monotonically increasing PSI with severity. jm1's one residual flagged-but-understood case at severity 0.25 is the only non-trivial validation-failure outcome in the corpus.

## 7. Module 5: `validity_violations.py` (D1 §8)

**Mechanism:** A cell's value is replaced with an invalid value for a proportion of selected cells, where the invalid value is chosen per-column from a one-time, human-confirmed domain-constraint annotation (no_negative: negate; range meaning a bounded interval: push beyond whichever bound; none, meaning no documented constraint: push beyond a train-fold IQR-based bound; categorical: excluded entirely, not a numeric magnitude constraint).

### Design notes

- This module's build was blocked for several turns pending D1 Decision Log item 6 (a one-time domain-constraint annotation pass, owned by Kishore) and a separate naming inconsistency (road-safety versus road_safety duplicate entries in an earlier draft of the annotation file). Both were confirmed resolved directly by inspecting the actual uploaded annotation file: all 17 datasets present, zero blank confirmations across 369 rows, and only the canonical road-safety name present. 175 of 369 rows (47%) showed the confirmed constraint type differing from the auto-suggested one — strong evidence the annotation pass was a genuine manual review, not a rubber stamp. Notably, EEG's channels were corrected from the auto-guessed no_negative to none, with the explicit note that observed non-negativity was a sampling artifact, not a real device constraint — directly consistent with what real-data testing of `missingness.py` had already suggested about EEG's channel behavior.
- The no_negative violation includes a guard: if negating a value would still not produce something below zero (for example, a pre-existing negative value in an otherwise-positive column), the value is pushed further negative rather than silently producing a non-violation.
- Two new D3 constants, both flagged as undocumented by D1: `IQR_MULTIPLIER` at 3.0 (deliberately more extreme than the standard 1.5x mild-outlier convention, since the injected value needs to read as implausible, not just statistically unusual) and `RANGE_OVERSHOOT_FRACTION` at 0.10.

### Testing journey

This is the one module that passed its first real-corpus run cleanly — no bugs found. Verified directly against all 17 datasets' actual constraint sets from the real annotation file, not just a synthetic subset, confirming correct parsing of every real constraint format, correct exclusion of categorical columns, and genuinely constraint-violating injected values (for example, latitude pushed to exactly plus or minus 108, outside the range of minus 90 to 90 by the configured overshoot).

### Final real-corpus result

255/255 OK. 253/255 validation_passed=True; the 2 misses (Airbnb, MagicTelescope) were traced to 3.5 to 3.8-sigma deviations, and notably landed on exactly the two datasets with the most columns checked per call (37 and 10 respectively), consistent with a multiple-comparisons effect rather than any defect: testing more columns simultaneously mechanically raises the chance that some column randomly crosses a fixed-confidence threshold.

## 8. Module 6: `uniqueness.py` (D1 §9)

**Mechanisms:** Global Duplication (a computed count of rows sampled with replacement, appended as exact duplicates, no jitter), Slice-Conditional Duplication (same mechanics, restricted to the frozen slice S).

### Design notes

- Structurally unique among all six modules: the corrupted output is larger than the input, since rows are appended, not transformed in place. Every other mechanism preserves row count.
- validation_passed here is deterministic, not statistical — the appended count is computed directly, not achieved via independent per-row Bernoulli draws the way rate-based mechanisms elsewhere are. A clean run means the row math held exactly, not "close enough within tolerance."
- Baseline versus injected duplicate rate are deliberately kept separate in the metadata: several corpus datasets (jm1, heloc) have documented natural duplicates from Phase 0, and generic post-hoc duplicate detection on the corrupted output would conflate pre-existing duplicates with newly-injected ones, overstating what the mechanism actually did.
- The slice-conditional variant obtains the frozen slice S from `base.py`'s shared function — the identical function `distribution_shift.py`'s concept shift uses — rather than computing its own.

### Testing and cross-mechanism verification

No bugs found — the module's core logic (direct count computation, exact row appending) left little room for the kind of subtle statistical or NaN-propagation bugs found elsewhere. The one property that mattered most to verify wasn't correctness of this module in isolation, but its alignment with `distribution_shift.py`: called both modules independently with identical inputs and confirmed byte-identical slice bounds, down to the exact floating-point value. This is the structural guarantee D1 §7.3/§9.2 requires (so D11's stratified evaluation can ask whether a region degrades under concept shift and duplication, or only one), confirmed working rather than assumed.

### Final real-corpus result

510/510 OK (17 datasets times 2 mechanisms times 5 severities times 3 seeds). Real natural-duplicate-rate measurements captured for all 17 datasets as a side effect of the runner's diagnostic logging — including two new findings not referenced in D1's original HELOC-only anchor: assistments at 13.68% and jm1 at 11.39%, both higher than HELOC's own confirmed 5.96%.

## 9. Cross-Mechanism Guarantees

Two structural guarantees were built into the shared infrastructure and explicitly verified, not just asserted:

1. **Frozen slice S sharing (D1 §7.3 and §9.2).** The shared slice function in `base.py` is the single source of truth. Verified: two independent callers (simulating `distribution_shift.py` and `uniqueness.py`) given identical inputs receive byte-identical slice bounds. Different seeds produce different slices (confirmed). Repeated calls with the same inputs are fully deterministic (confirmed across 5 repetitions).
2. **Shared rate, variance, and NaN-handling logic (`missingness.py` and `feature_noise.py`).** The rate-tolerance, reference-variance, and NaN-adjustment helpers were moved to `base.py` once both modules needed identical logic, rather than maintaining two copies that could silently drift apart over time.

## 10. Runner Scripts

Six runner scripts, one per module, all following the same structure: discover processed datasets, split train and test (70/30, stratified by target, seed 42 — itself a new D3 decision, since D2 established the principle of splitting before corrupting but not a ratio), run every combination of mechanism, placeholder severity, and seed, log live progress, write a manifest CSV, print a summary.

## 11. Testing Methodology

Every module went through the same sequence, without exception:

1. **Synthetic construction** covering the mechanism's core behavior and known edge cases (pre-existing NaN, constant columns, low-cardinality categories, class-emptying severities).
2. **Direct verification of mathematical properties**, not just whether it ran — for example, confirming PSI actually rises monotonically with severity, confirming class-conditional noise's directional rates actually respect the target asymmetry ratio, confirming negated values are genuinely negative.
3. **Real-corpus validation** via the runner scripts, against all 17 actual processed datasets.
4. **Root-cause diagnosis before any fix** — every bug was reproduced synthetically first, with the specific data property responsible isolated and confirmed, before writing a fix. Two false starts are part of this record deliberately, not hidden: an initial claim of a real, reproducible RNG bias in missingness was tested against 500 seeds and retracted; an initial near-constant-column diagnosis for EEG's MNAR failure was directly disproven by the real correlation data before the actual cause (all-or-nothing blocking design) was found.
5. **Re-verification against real data after every fix**, not just synthetic re-testing — confirming the specific real dataset that surfaced the bug now behaves correctly, in addition to confirming no regression in previously-passing cases.

## 13. Real-Data Findings (Summary)

Full detail in the companion decisions and findings document. Headlines:

- **EEG**: genuine, largely independent channel structure — most cross-channel correlations near zero, AF3 uniquely weak and borderline in its self-dependency, F7 reliably strong. Independently corroborated by the annotation file's own correction of EEG's channels from no_negative to none.
- **jm1**: correlated, outlier-heavy software-complexity features cause both a fixed covariate-shift degeneracy and an understood, residual effective-sample-size reduction at high severity, plus the second-highest natural duplicate rate in the corpus at 11.39%.

## 15. What's Explicitly Not Done

- **D4's severity grid** does not exist yet — every runner uses an illustrative placeholder grid (0.05, 0.10, 0.15, 0.20, 0.25), not real values.
- **Per-dataset overrides** for the uncanonicalized constants (for example, a lower G_CLIP_Z ceiling specifically for jm1) have no implementation path yet — every constant is currently a single corpus-wide Python default.
