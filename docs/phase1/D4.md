# D4 — Severity Grid: Complete Build Report

**Project:** When Does Data Quality Predict Model Failure? A Benchmark for Knowing Before It Breaks
**Status:** Approved (David, 2026-07-20), v1.2. Severity table generated and independently verified against the real corpus.
**Companion documents:** `D4_severity_grid.md` (the decision record itself), `D3_implementation_decisions_and_findings.md` and `D3_complete_build_report.md` (D3's equivalent documents, which D4 depends on directly).

---

## 1. Overview

D4's job, stated precisely by D1 §10: turn D1's calibration _methodology_ into actual _numbers_. D1 names an anchoring principle per mechanism class (missingness → natural per-column rates, uniqueness → natural duplicate rates, validity → natural sentinel/constraint violations, label noise → Cleanlab-estimated error rate) but deliberately stops short of numeric values, delegating that to D4. D1 §14 additionally hands D4 the experiment-budget arithmetic (47,600 rows), explicitly flagged as "illustrative" and D4's to adjust.

Concretely, D4 needed to produce a specific numeric severity value for every `(dataset, sub-mechanism, severity-level 1–10)` cell — replacing the placeholder grid `[0.05, 0.10, 0.15, 0.20, 0.25]` D3's six runners had been using for smoke-testing only.

D4 went through three distinct rounds, each triggered by contact with real data:

1. **Initial draft** — a governing rule (linear severity spacing) plus five open decisions, two elicited directly, three resolved decisively.
2. **Real-data investigation** — running a profiling script against the actual corpus overturned the original per-dataset anchoring plan for two of six mechanism classes.
3. **Real-data investigation, round two** — running the actual severity-table generator surfaced a serious methodology failure in the label-noise anchor, caught and fixed before it could propagate.

This mirrors D3's own pattern closely: design on paper, then let real data find what paper couldn't.

## 2. Round 1: Initial Draft

### 2.1 Governing rule

Adopted the simplest possible resolution consistent with D1's anchoring requirement: 10 severity levels, linearly spaced from near-zero to a mechanism-specific `α_max`.

```
severity(ℓ) = (ℓ / 10) × α_max,   ℓ = 1, ..., 10
```

Non-uniform spacing (denser near known behavioral transitions) was considered and explicitly deferred to Future Work rather than adopted — the simple rule was judged sufficient for this version.

### 2.2 Two elicited decisions

Two open items were judged genuine research-design trade-offs, not engineering defaults, and were put to a direct choice rather than resolved unilaterally:

- **jm1's covariate-shift severity ceiling.** D3 had already found and fixed a degenerate-resample bug for jm1 (see D3's build report), but a milder residual remained: effective sample size collapsing to 4.6% at severity 0.25 specifically. Three options were offered (cap jm1's ceiling below this point / keep the full grid and disclose the limitation / keep the full grid and flag it programmatically in D7's manifest). **Chosen: flag in the manifest** — reuses diagnostic fields D3 already computes, avoids building per-dataset severity-override infrastructure, and is more protective than a passive disclosure since it's programmatically checkable.
- **Uniqueness severity anchoring philosophy.** Natural duplicate rates ranged from ~0% to 13.68% (assistments) across the corpus. Three options offered (per-dataset anchor / single corpus-wide anchor / hybrid with floor-ceiling). **Chosen: single corpus-wide anchor**, using HELOC's confirmed natural rate — simplest structurally, and on inspection, more textually faithful to D1 §9's own wording ("HELOC's 588 duplicate rows anchor α_max," a singular, corpus-wide framing) than a per-dataset scheme would have been.

### 2.3 Three decisively-resolved items

- **Quantization's `k_max`**: confirmed at 20 (a decision already made in an earlier session, carried forward).
- **`SLICE_FRACTION = 0.30`**: confirmed unchanged from D3's implementation default.
- **Label-shift's `rho_target`**: frozen (no longer a D3-era placeholder) at 0.9/0.1 toward the farther extreme from a dataset's natural prior, with an initial safety floor of "20 rows or 2% of fold, whichever is larger."

### 2.4 What Round 1 left open

Two sections — missingness's and validity's `α_max` — were written as _formulas only_, with the actual numeric values flagged as unresolved: D1 only cites one example value each (road-safety's ~8.96% missingness, HELOC's sentinel story), and no full 17-dataset table existed anywhere in the project yet.

## 3. Round 2: Real-Data Investigation Overturns the Original Anchoring Plan

A profiling script (`profile_natural_defect_rates.py`) was built specifically to pull the two missing tables — natural per-column missingness rates and natural constraint-violation rates — for all 17 datasets, reusing `validity_violations.py`'s own constraint-parsing logic so the profiling couldn't drift from what the corruption engine actually implements.

### 3.1 The finding that forced a redesign

Running it against the real corpus revealed the original "per-dataset own-rate anchor" plan was not just inelegant but **non-functional for most of the corpus**:

- **11 of 17 datasets (65%) show zero natural missingness.**
- **15 of 17 datasets (88%) show zero natural validity violations.**

A per-dataset anchor would set `α_max = 0` for those datasets — every severity level from 1 to 10 injecting nothing, for the majority of the corpus, on two of six mechanism classes. This wasn't a rough edge to smooth over; the per-dataset approach as originally designed simply didn't work.

### 3.2 The fix, and why mean (not max) was the right statistic

Switched both mechanisms to a single corpus-wide anchor — the same philosophy already chosen for uniqueness in Round 1 — using the specific datasets D1 itself already named as examples:

- **Missingness → road-safety's mean natural missingness rate, 8.65%.** This is a near-exact match to D1 §10's own cited figure ("road-safety's ~8.96%"), which directly validated **mean** over **max** as the correct per-column summary statistic. road-safety's own max was 47.73% (`2nd_Road_Class`, conditionally missing by design for certain junction types, not a general defect rate) — using max anywhere in the corpus would have been badly distorted by single pathological columns. Two other datasets confirmed this starkly: `kick`'s max was 95.24% (`PRIMEUNIT`, essentially never populated) and `diabetes_readmission`'s max was 96.82% (`weight`, a well-documented near-total-missingness field in this exact dataset).
- **Validity → HELOC's mean natural constraint-violation rate, 10.88%.** Only 2 of 17 datasets showed any natural violation at all (`bank-marketing`, `heloc`) — again ruling out per-dataset anchoring outright, and again pointing directly at D1's own named example.

### 3.3 Two new findings, worth citing alongside HELOC's original sentinel story

- **`bank-marketing`'s column `V14`** (almost certainly `pdays`, given its position in the feature ordering) showed an **81.74%** natural violation of its `no_negative` constraint — consistent with `pdays`'s well-documented `-1` sentinel convention for "never previously contacted in a prior campaign," which applies to the large majority of this dataset's rows. A textbook real-world sentinel pattern, not a data error.
- **HELOC's sentinel structure, now measured precisely, is two-tiered.** 13 of its 23 constrained columns share an identical 5.62% baseline rate — consistent with a shared sentinel code hitting applicants with no credit history at all across many fields simultaneously. A handful of feature-specific columns run much higher: `MSinceMostRecentDelq` at 51.9%, `NetFractionInstallBurden` at 38.3%, `MSinceMostRecentInqexcl7days` at 27.9% — sensible, since "never delinquent" or "never had an install-burden account" are each much larger populations than "no credit history at all."

### 3.4 Document revision

D4 §2.1 and §2.5 were rewritten with the resolved formulas and real values; Decision Log entries 2 and 9 updated; Future Work revised to note that per-dataset anchoring for missingness/validity was _actively investigated and ruled out_, not merely deferred — a distinction from uniqueness's per-dataset anchoring, which remains a live option since its natural rates are mostly non-zero and reasonably distributed.

## 4. David's First Review Round

Three items were flagged for explicit sign-off, since they carry real trade-offs rather than obviously-correct defaults:

1. **jm1's covariate-shift ceiling** — _"Keep the uniform grid. Flagging it in the manifest is the right call — we want jm1 in the corpus for the same reason we kept the weak-effect uniqueness regime: a hard case is more informative than a convenient one we've engineered away."_ **Approved as documented.**
2. **The uniqueness anchor** — _"Fine for this version. It underrepresents assistments'/jm1's natural duplicate structure, but that's an honest, disclosed trade-off for simplicity... only revisit per-dataset anchoring if Phase 2 results show the corpus-wide choice is actually hiding something."_ **Approved.**
3. **The label-shift safety floor** — _"A 20-row absolute floor is an odd number to defend if anyone asks where it came from — it reads as made up, because it is. Drop the absolute-row alternative, keep it simple: a flat 5% of the fold, nothing else."_ **Approved with a revision** — the mixed absolute/relative floor was simplified to a flat 5%, incorporated into Decision 7.

D4 was marked Approved (v1.1) at this point — before the second, more serious real-data finding below.

## 5. Round 3: The Severity Table Generator Finds a Second Real Problem

With D4's formulas finalized, a generation script (`generate_severity_table.py`) was built to actually compute the 17×14×10 table. Most sub-mechanisms needed no data pull at all — 11 of 14 use D4's fixed corpus-wide constants directly. Two categories needed real computation: label noise's per-dataset `α_max` (originally specified as a Cleanlab-estimated overall label-error rate, per D1 §6.2 and D4's original Decision 5), and distribution shift's label mechanism, which needs each dataset's natural class prior to set `rho_target`.

### 5.1 The finding

Running the generator against the real corpus produced label-noise `α_max` estimates ranging **13% to 56%** across the 17 datasets, with EEG's raw estimate at **55.74%**.

These are not plausible label-noise rates for curated benchmark datasets. The decisive piece of evidence: **Phase 0 had already screened every one of these 17 datasets for learnability via XGBoost 5-fold cross-validation, requiring reasonable AUC to pass the corpus at all.** A dataset with genuinely 40–55% mislabeled examples could not have shown reasonable learnability under that screen. The Cleanlab estimate and the Phase 0 screen directly contradicted each other, and the Phase 0 screen — an established, already-validated measurement — was judged the one to trust.

### 5.2 Diagnosis

Cleanlab's confident-joint method interprets "the probe model disagrees with this label" as evidence of a label error. The probe used was plain logistic regression — a linear model. If a dataset's true decision boundary is non-linear (plausible for several datasets in this corpus, especially EEG), the probe will disagree with plenty of _correctly_-labeled points too, inflating the estimated noise rate. This is a known failure mode of confident learning when the probe underfits, not a coding bug — the estimation ran correctly and reproducibly; the method itself was producing a biased answer.

No simple fix was available. A stronger probe (gradient boosting, a larger MLP) would likely reduce the bias, but three of the four model families this project evaluates are exactly the "stronger probe" candidates — using one would reintroduce the probe/evaluated-family circularity D1 §6.3 already warns about for a related mechanism (boundary-aware label noise), which is precisely why that mechanism was deferred in the first place.

### 5.3 The fix

**Reverted label noise's `α_max` from a per-dataset Cleanlab estimate to a single fixed corpus-wide default: 0.10 (10%).** This mirrors the same pattern missingness, validity, and uniqueness had already converged on for different but related reasons — in those cases, insufficient natural signal; here, an unreliable estimation method. The 10% figure is explicitly labeled a stopgap, not a measured anchor.

Class-conditional label noise's _asymmetry ratio_ (a separate parameter from the overall rate, still estimated per-dataset by `label_noise.py` itself) was not affected by this change — a ratio between two off-diagonal counts is less sensitive to overall probe confidence than an absolute rate estimate is, and this parameter's estimation wasn't implicated in the same failure mode.

### 5.4 David's second review

_"Don't use these numbers — they don't pass a basic sanity check against Phase 0's own screening. Freeze label noise at a flat 10% corpus-wide for this version, flag the estimation methodology as broken, and put fixing it in Future Work. We're not going to build Phase 2 on a noise-rate assumption we don't trust."_ **Approved with this revision**, explicitly recorded as reversing part of the original Round-1 approval rather than silently changed. D4 moved to v1.2.

### 5.5 Script revision

`generate_severity_table.py` was revised to remove the Cleanlab-based estimation function entirely (rather than leave dead code guarded by a flag), switching label noise to the same `corpus_wide` code path every other fixed-`α_max` mechanism already used. Re-tested: confirmed `label_noise` severity at level 5 now computes to exactly `0.05` (`(5/10) × 0.10`), with no Cleanlab dependency in the run path at all — faster and simpler than the version it replaced, not just more honest.

## 6. Final Validated Severity Table

Re-run against the full real corpus and independently verified (not just trusted from the console log):

- **2,380 rows** (17 datasets × 14 sub-mechanisms × 10 severity levels — exact).
- **Zero nulls** in `severity_value`.
- **Every corpus-wide mechanism resolves to exactly one unique value at severity level 10**, matching D4's `α_max` precisely:

| Sub-mechanism(s)               | `α_max` at level 10                                  |
| ------------------------------ | ---------------------------------------------------- |
| `mcar`, `mar`, `mnar`          | 0.0865                                               |
| `gaussian`, `categorical_swap` | 0.30                                                 |
| `quantization`                 | 1.0 (severity is the direct k-range fraction)        |
| `uniform`, `class_conditional` | 0.10                                                 |
| `covariate`                    | 0.25                                                 |
| `label`                        | 1.0 (severity is D1's direct α interpolation weight) |
| `concept`                      | 0.5                                                  |
| `validity_violations`          | 0.1088                                               |
| `global`, `slice_conditional`  | 0.06                                                 |

`distribution_shift/label` additionally carries a per-dataset `rho_target` (0.9 or 0.1, clamped to the 5% floor), computed directly from each dataset's natural class prior — the one genuinely per-dataset piece of information in the table besides the (now-abandoned) label-noise attempt.

## 7. What's Still Open

Carried forward explicitly in D4's Future Work section, not silently dropped:

1. Uniqueness's corpus-wide anchor (unlike missingness/validity) remains a live candidate for per-dataset refinement — its natural rates didn't force the same reversal.
2. A hybrid per-dataset/corpus-wide anchor for missingness and validity, using non-zero datasets' own rates and falling back to the corpus anchor only at zero, was considered and explicitly not adopted for this version.
3. HELOC's validity anchor (10.88%) blends a 5.62% baseline with several much higher feature-specific rates — a more precise model could separate these rather than average them.
4. The label-shift minority floor (5%) has no grounding beyond "a clean, defensible round number" — needs revisiting once D11's deferred conformal-interval cell-size requirements actually exist.
5. Feature noise has no natural anchor at all (D1 §10 doesn't name one for this mechanism class) — its 0.30 ceiling is asserted, not measured.
6. `online_shoppers`' outlier asymmetry ratio (6.73, from D3's testing) hasn't been independently re-examined.
7. **The label-noise estimation methodology itself needs to be fixed** — this is real, unstarted work, not a nice-to-have, if label noise's severity axis is meant to eventually reflect real per-dataset signal rather than a flat placeholder.

## 8. Process Note

D4's build followed the same discipline D3's did throughout: state a formula, then test it against real data before trusting it. Both real-data rounds in this document found something a purely paper-based design process would have missed — the zero-rate problem in Round 2, and the probe-underfitting problem in Round 3. Neither was visible from D1's text or from D4's own internal logic; both only became visible once actual numbers were computed and sanity-checked against an independent, already-established measurement (D1's cited examples in Round 2, Phase 0's learnability screen in Round 3). This is the same lesson D3's build report closes with, reconfirmed on a different deliverable.
