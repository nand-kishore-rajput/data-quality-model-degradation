# SECTION 1 — Benchmark Execution Framework

## 1.1 Purpose

This section documents the operational conventions that govern how every corruption experiment is discovered, executed, logged, and recovered from failure — as actually implemented and validated across all six D3 mechanism runners. It formalizes decisions already made in code, rather than proposing new ones.

## 1.2 Dataset Discovery

Every runner uses an identical `discover_datasets()` function: walk `datasets/processed/`, treat each immediate subdirectory as a `source` (`cleanml`, `openml`, `uci`, `folktables`, `tableshift`), and pair every `.parquet` file found with `(source, dataset_name, path)`. This is the single, shared mechanism by which any script determines what the corpus actually contains — no runner hardcodes the 17-dataset list independently; each rediscovers it from the filesystem at runtime.

**Decision:** `--all` runs every discovered dataset; `--datasets X Y Z` filters to a named subset; no flag defaults to the first 3 discovered (a smoke-test convenience, not used in any full-scale run documented in D3/D4).

## 1.3 Directory Structure and Output Convention

**Decision:** all six runners write to one shared directory, `/workspace/results/corruption_engine/`, using the naming pattern:

```
{mechanism}_smoketest_manifest.csv   -- the per-mechanism corruption metadata record
{mechanism}_smoketest_run.log        -- the corresponding human-readable log
```

D4's `severity_table.csv` and D7's eventual `manifest.csv`/`manifest_clean_baselines.csv` live in the same directory, making `results/corruption_engine/` the single source of truth for every Phase 1 corruption-related artifact. (The `_smoketest_` naming segment is a historical artifact of these scripts' original smoke-test purpose; the schema and behavior are identical at full scale, per D7's confirmed verification — renaming the output files was considered and not adopted, since doing so would require re-pointing every downstream reference for no functional benefit.)

## 1.4 Logging Convention

**Decision:** a shared `log(msg, end="\n")` helper writes to both stdout (for live monitoring) and the run's log file simultaneously, with the file handle opened in line-buffered mode (`buffering=1`) so progress is visible in the log file in real time, not only after the process exits. Every dataset's section is delimited by a fixed-width separator line and a `[HH:MM:SS]` timestamp, making it possible to scan a multi-hour full-scale log for exactly where a given dataset started and how long it took.

## 1.5 Failure Recovery — the `status` Taxonomy

**Decision:** every corruption call is wrapped so that a failure at any level (missing target-column mapping, unreadable file, non-binary target, mechanism-specific corruption error, or a genuinely unexpected exception) is caught, recorded as a row with a descriptive `status` value, and logged — but never allowed to halt the run for other datasets or other severity/seed combinations. Confirmed status vocabulary, observed across all six full-scale runs:

| Status                                               | Meaning                                                                            | Fatal to the run?                                                              |
| ---------------------------------------------------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `OK`                                                 | Corruption succeeded, fidelity check passed                                        | No                                                                             |
| `PARTIAL_SUCCESS`                                    | Some requested columns excluded, others corrupted successfully                     | No — expected, informational                                                   |
| `SKIPPED_NO_TARGET_MAPPING` / `TARGET_COL_NOT_FOUND` | Dataset registry gap                                                               | No — logged, dataset skipped                                                   |
| `SKIPPED_NO_ELIGIBLE_COLUMNS`                        | No columns of the required type exist for this mechanism                           | No — logged once per mechanism, not once per severity/seed                     |
| `NON_BINARY_TARGET_{n}`                              | Target has other than 2 classes (label/concept shift, label noise are binary-only) | No                                                                             |
| `CLEANLAB_UNAVAILABLE`                               | Cleanlab dependency missing (class-conditional label noise only)                   | No — uniform-noise rows still complete                                         |
| `NOT_IMPLEMENTED`                                    | Deferred mechanism called (boundary-aware label noise, D1 §6.3)                    | No                                                                             |
| `ALL_COLUMNS_FAILED`                                 | Every requested column failed its own mechanism-fidelity check                     | No — logged as a genuine investigate-worthy case                               |
| `CORRUPTION_ERROR`                                   | A caught `ValueError`/`RuntimeError` from the corruption engine itself             | No                                                                             |
| `UNEXPECTED_ERROR: {type}`                           | An uncaught exception class, with the error message truncated and preserved        | No — this is the category that surfaced the missingness bool-column bug (§1.7) |

**This taxonomy is the project's actual failure-recovery design.** No global try/except wraps the whole run; recovery happens at the smallest unit (one `dataset × mechanism × severity × seed` call), so a single bad column or dataset never costs the rest of the corpus.

## 1.6 Reproducibility — Deterministic Per-Call Seeding

**Decision:** every random operation — corruption injection and the train/test split alike — derives its actual seed via `derive_call_seed(seed, dataset_name, mechanism, severity)` (SHA-256 hash of the full call context), never by passing the caller's raw seed index directly into `np.random.default_rng()`. This is not a stylistic preference; it is the fix for a confirmed, quantified bug (D3 Era 2, §5.3 of the build report): reusing a small, fixed seed pool directly across many calls produced correlated, non-independent draws, discovered via a 5.65×10⁻⁶-probability statistical anomaly. `derive_call_seed` is applied uniformly across all six modules and all six runners' `split_train_test` functions, giving the whole framework one, single-source-of-truth reproducibility guarantee: identical `(seed, dataset, mechanism, severity)` always regenerates a byte-identical corrupted fold, on any machine, at any time — the property D7's generator design (§5) depends on directly, since it regenerates corrupted folds on demand rather than persisting them (§1.8).

## 1.7 Incremental Persistence and Resumability

**Decision:** every runner calls `pd.DataFrame(results).to_csv(OUT_MANIFEST, index=False)` after completing each dataset, not only once at the end of the full run. This means a crash partway through a 17-dataset run loses at most the in-progress dataset's rows, not the entire run — confirmed as the operative resumability mechanism for this framework (as opposed to caching corrupted data itself, which is deliberately not done — see §1.8).

**Real-world validation of this framework, end to end:** all six runners were executed at full scale (17 datasets × full D4 severity grid × 5 seeds) during Phase 1 closure. One genuine bug was found and fixed via this framework's own mechanisms — `online_shoppers`' boolean `Weekend` column, surfaced as `UNEXPECTED_ERROR: TypeError` rows in `missingness`'s manifest, diagnosed from the preserved `error` field, fixed in `eligible_columns()`, and confirmed resolved on rerun (0 errors, exact expected row count). This is the failure-recovery and logging design (§1.5, §1.4) working exactly as intended — a real defect surfaced, isolated, and fixed without needing to rerun anything but the one affected mechanism.

## 1.8 What Is Deliberately Not Persisted

**Decision, confirmed directly in all six runners' code (`_, meta = corrupt(...)`):** corrupted DataFrames themselves are never written to disk. Only the metadata (`CorruptionMetadata` — achieved rates, validation flags, diagnostics) is persisted. This is a deliberate design choice, not an oversight: because `derive_call_seed` (§1.6) makes every corruption call fully deterministic, any corrupted fold can be regenerated on demand rather than cached — avoiding tens of gigabytes of redundant storage across the full 46,800-row experiment grid, and avoiding the staleness risk of a cached fold silently surviving a later code fix (the same risk `base.py`'s frozen-slice-cache design explicitly reasons about). D7's `generate_phase2_manifest.py` is built directly on this guarantee: it calls `corrupt()` itself, in-process, rather than looking for a corrupted-data file that does not and should not exist.

## 2.4 Findings

### 2.4.1 — kick's high cardinality is legitimate, not a defect

`Model` (999 categories), `SubModel` (799), `VNZIP1` (150), and `Trim` (129) are all high-cardinality but genuinely legitimate — this is the well-known used-car-auction dataset, and hundreds of distinct models is exactly what real inventory data looks like.

**Actionable for D7/Phase 2:** naive one-hot encoding of these four columns together would explode into thousands of dummy columns. kick will need deliberate categorical-encoding treatment (target encoding, hashing, or a cardinality cap) that most of the other 16 datasets won't. _(Confirmed still relevant: kick's feature_noise full run showed 20 categorical columns, including `Model`/`SubModel`/`Trim`/`VNZIP1`, all handled without error by `categorical_swap` — but this says nothing about Phase 2's eventual encoding strategy, which remains an open decision.)_

### 2.4.2 — EEG's skew signature is sharper than earlier characterized

Per-channel skew/std:

| Channel Group                 | Skew                       | Std       |
| ----------------------------- | -------------------------- | --------- |
| FC5, O1, P7, P8, AF3, F8, AF4 | 118–122 (nearly identical) | 1208–5891 |
| O2, F7, FC6, F4, T8, T7       | 7.6–51                     | 29–46     |
| F3                            | −13.6 (only negative)      | 44        |

Seven channels — not just AF3 — share an almost identical, extreme skew signature (~118–122). This is too tight a cluster to be coincidental; it points to a shared artifact event (e.g., simultaneous electrode saturation or a movement artifact affecting several electrodes at once), rather than one channel behaving badly in isolation.

This refines D3's earlier bug-fix narrative, which correctly identified AF3 as a problem channel but characterized it as uniquely bad. The corrected framing: **"EEG has a coherent multi-channel artifact signature affecting 7 of 14 channels," not "AF3 is uniquely problematic."**

F3's negative skew is a distinct pattern — the only channel with a heavy low-side tail rather than high-side.

**Action item:** D3's build report currently still says "AF3" singular — needs a dated amendment to reflect this finding. _(Still open — confirmed not yet actioned as of this merged document.)_

### 2.4.3 — Scale ratio alone does not predict the covariate-shift collapse bug; skew concentration does

Airbnb has the highest scale ratio in the corpus (18,232,377 — 7x jm1's 2,705,693, the dataset that actually broke in D3), yet was never flagged as problematic.

jm1 and EEG both combine extreme scale disparity with **concentrated skew** (20/21 and 14/14 highly-skewed columns respectively); Airbnb's skewed-column proportion is much lower (10/38).

**Conclusion:** large scale ratio is a necessary-but-insufficient warning sign — the actual failure mode needs concentrated skew too. road-safety (2,105,717 scale ratio, 31/61 skewed — the highest skewed-column _count_ in the corpus) shares both risk factors and is worth a targeted covariate-shift check even though it wasn't the dataset that originally surfaced the bug. _(Confirmed clean on the full-scale run — road-safety's covariate shift showed monotonic PSI with no degenerate-resample flag, consistent with the Era 2 MAD-based clipping fix generalizing correctly.)_

### 2.4.4 — assistments is the most data-quality-challenged dataset on two independent axes simultaneously

Highest missingness (12.4% overall / 56.5% max column) **and** highest duplication (30.4%). No other dataset tops both dimensions.

### 2.4.5 — phoneme is the corpus's only near-perfectly-scaled dataset

Scale ratio ≈ 1.0000000176 — a clean contrast case against jm1/EEG/road-safety's extreme heterogeneity.

### 2.4.6 — Duplicate-rate measurement scope must be stated explicitly

assistments' full-dataset duplicate rate (30.4%, measured here) is more than double the test-fold-only rate found in D3 (13.68%) and the full-scale uniqueness runner's own measurement (14.28%, §1.9's underlying data). Not a contradiction — different populations: the full-dataset measurement catches cross-fold duplicate pairs that a single-fold check structurally cannot see.

**Any future citation of a dataset's duplicate rate must specify which population it was measured against.**

## 2.5 Implications for D7

- kick needs an explicit categorical-encoding decision before Phase 2.
- road-safety deserves a targeted covariate-shift check, despite not being the dataset that originally surfaced the bug — **now confirmed clean** (§2.4.3).
- D7's manifest must record which population any "duplicate rate" field was measured against (full dataset vs. test-fold-only).
- assistments' and diabetes_readmission's missingness, and assistments' duplication, are citable data-quality findings — same category as HELOC's sentinel story and bank-marketing's `pdays` convention.
