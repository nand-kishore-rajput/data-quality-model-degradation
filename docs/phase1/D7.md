# 1. Purpose

D7 is the single artifact turning six independently-run, now fully-validated corruption manifests, D10's metrics taxonomy, D4's severity grid, and a model-evaluation loop into **one row per training sample** — exactly what D11's meta-model consumes and D12's attribution pipeline groups by fold and model family.

This document has two parts: **the schema** (§4, finalized and verified) and **the generator design** (§5–§8, fully specified, not yet implemented). Nothing in this document is speculative — every design decision below either reuses a convention already validated in D6's Execution Framework or corrects a specific, real discrepancy found during full-scale verification.

# 2. Row Granularity and Corrected Expected Size

**Decision (unchanged, re-confirmed):** one manifest row = one `(dataset, sub_mechanism, severity_level, seed, model_family)` combination.

**Corrected expected total: 46,800 rows, not 47,600.**

D1 §14's original 47,600 figure (17 × 14 × 10 × 5 × 4) assumes every `(dataset, sub_mechanism)` pair is structurally valid. Full-scale execution confirmed **four pairs are not**:

| Dataset        | Sub-mechanism      | Why impossible                                            |
| -------------- | ------------------ | --------------------------------------------------------- |
| EEG            | `categorical_swap` | Zero categorical columns (all 14 channels are continuous) |
| MagicTelescope | `categorical_swap` | Zero categorical columns                                  |
| jm1            | `categorical_swap` | Zero categorical columns                                  |
| phoneme        | `categorical_swap` | Zero categorical columns                                  |

Each excluded pair removes `10 severities × 5 seeds × 4 model families = 200` rows from the total: `47,600 − (4 × 200) = 46,800`.

**Consequence for the generator:** it must pre-check column-type eligibility for `feature_noise`'s three sub-mechanisms _before_ attempting `corrupt()`, using the same classification logic already validated in `run_feature_noise_corruption.py`'s `classify_columns()` — and treat these four combinations as a structural exclusion, asserted and logged, not a failure mode. **D8's validation report should assert `manifest.csv` has exactly 46,800 rows and explicitly document these four exclusions as expected, not investigate them as gaps.**

# 5. Assembly Pipeline — Regeneration, Not Loading

**Confirmed, load-bearing fact (D6 §1.8):** no corrupted DataFrame is ever persisted anywhere. All six manifests are validation records only. The generator must call `corrupt()` itself, in-process, for every row — never attempt to read a corrupted fold from disk.

```
Step 1: For each dataset x model_family (17 x 4 = 68 combinations):
        Train once on the clean train fold -> m_clean
        Store in manifest_clean_baselines.csv (68 rows, auditable, joined
        in later rather than duplicated inline 46,800 times with no
        single source of truth)

Step 2: Build the iteration list from severity_table.csv's 2,380 rows x 5
        seeds = 11,900 (dataset, sub_mechanism, severity_level, seed)
        combinations. Before adding a combination to the list, apply the
        column-eligibility pre-check (Section 2) for feature_noise's three
        sub-mechanisms -- exclude the 4 known-impossible pairs (800 rows
        at the final, per-model-family count) explicitly, logged as a
        structural exclusion, not attempted and caught as a failure.

Step 3: For EACH of the (now) 11,700 valid combinations:
        a. Regenerate the corrupted fold: call corrupt() directly, using
           the SAME (dataset, mechanism, severity_value, seed) as the
           corresponding row in that mechanism's already-validated D3
           manifest -- derive_call_seed (D6 Section 1.6) guarantees this
           reproduces an identical corrupted fold.
        b. Check the returned meta.status / meta.validation_passed --
           if not OK or PARTIAL_SUCCESS, log and skip to step (f), writing
           a manifest row with status preserved and all metric/performance
           columns NaN (never silently drop the row -- D10 Decision 12's
           NaN-not-zero-not-dropped principle applies here too).
        c. Compute D10's 20 metrics (clean train fold vs. this regenerated
           corrupted fold).
        d. Loop over the 4 model families: load Step 1's pre-trained model,
           evaluate on this SAME regenerated corrupted fold -> m_corrupted,
           compute relative_delta_m. (The corrupted fold is regenerated
           ONCE per combination, not once per family -- see Section 5.1.)
        e. Append one manifest row per model family (4 rows per
           combination).
        f. Continue to the next combination.

Step 4: Join everything into manifest.csv. Assert exactly 46,800 rows
        (11,700 valid combinations x 4 families). Assert zero duplicate
        manifest_row_id values.
```

## 5.1 Loop-Ordering Optimization (Explicit, Not Left Implicit)

**Decision:** the corrupted fold for a given `(dataset, sub_mechanism, severity, seed)` is identical across all 4 model families — only the model being evaluated changes. The generator must regenerate each corrupted fold **once** (11,700 times total) and loop over the 4 families only for the evaluation step, not regenerate the same fold 4 times redundantly (which would mean 46,800 corruption calls instead of 11,700 — a 4x, entirely avoidable cost). This was flagged as a risk in earlier design discussion and is stated here as a hard requirement, not a suggestion.

## 5.2 Resumability

**Decision, reusing D6 §1.7's already-validated convention:** write `manifest.csv` incrementally — after each `(dataset, sub_mechanism)` block of combinations completes (170 blocks total: 17 datasets × ~10 sub-mechanisms, accounting for the exclusions), not only at the very end. On restart, check which `(dataset, sub_mechanism, severity_level, seed, model_family)` keys already have a row and skip them. Corrupted folds themselves are never cached (§5, D6 §1.8) — only the generator's own row-level progress is checkpointed, exactly mirroring the six runners' own `to_csv`-after-each-dataset pattern.

# 6. Handling Non-`OK` Status Rows

**Decision:** a manifest row is generated for every valid `(dataset, sub_mechanism, severity_level, seed, model_family)` combination regardless of that combination's underlying `status`, but:

- **`OK` and `PARTIAL_SUCCESS`** rows get full D10 metrics and model evaluation computed.
- **Any other status** (`ALL_COLUMNS_FAILED`, `CORRUPTION_ERROR`, `UNEXPECTED_ERROR: *`, `CLEANLAB_UNAVAILABLE`, `NOT_IMPLEMENTED`) gets a row with `status` preserved, all D10/performance columns `NaN`, and is excluded from Phase 2 training by a `usable_for_training` boolean column (derived as `status in {"OK", "PARTIAL_SUCCESS"}`).

**Rationale:** dropping these rows silently would make the manifest's row count untraceable back to `severity_table.csv`'s enumeration; keeping them with `NaN` and an explicit usability flag preserves full auditability (consistent with D10 Decision 12's reasoning for a different column) without contaminating Phase 2 training with unusable data. **Confirmed low-impact:** at full scale, only feature_noise's 4 `SKIPPED_NO_ELIGIBLE_COLUMNS` rows (already excluded structurally per §2) and zero `UNEXPECTED_ERROR`/`ALL_COLUMNS_FAILED` rows exist across all six manifests — this handling path exists for robustness, not because it's heavily exercised in the current corpus.

# 9. Optimization Strategy (Summary of Implementation-Easy Decisions)

| Decision point          | Choice made                                                        | Why it's the easy-to-implement, low-risk option                                                                         |
| ----------------------- | ------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------- |
| Expected row count      | 46,800, not 47,600                                                 | Confirmed via full-scale execution, not assumed from D1's illustrative arithmetic                                       |
| Corrupted-fold handling | Regenerate via `corrupt()`, never cache                            | Deterministic (`derive_call_seed`), avoids storage bloat and staleness risk (D6 §1.8)                                   |
| Loop ordering           | Regenerate once per combination, evaluate across 4 families inside | Avoids a 4x redundant-computation mistake                                                                               |
| Resumability            | Incremental `to_csv`, key-based skip on restart                    | Directly reuses D6's already-validated six-runner convention, no new mechanism needed                                   |
| Non-OK status rows      | Kept with NaN metrics + `usable_for_training` flag                 | Full auditability without contaminating training data                                                                   |
| `status` column         | Added as new v2.0 schema field                                     | Confirmed necessary — without it, non-OK rows are indistinguishable from OK ones once metrics are NaN for other reasons |
| File format / location  | CSV, same shared directory                                         | Zero new dependencies, consistent with D6 §1.3                                                                          |

# 11. What Remains — the Actual Next Step

Everything in this document is design, verified against real data. The one thing that does not yet exist is the code: `generate_phase2_manifest.py`, implementing §5–§6 above. Concretely, that script needs:

1. **Model training utilities** for XGBoost, Random Forest, Logistic Regression, and MLP, applied consistently across all 17 datasets' clean train folds (Step 1, §5) — confirm whether these already exist in the codebase or need to be built as part of this script.
2. **A D10 metric-computation module** — `(reference_fold, corrupted_fold) -> 20-value vector`, per D10 §9's single-pipeline-entry-point requirement. This does not yet exist as code; only the specification (D10) exists.
3. **The main orchestration loop** itself (§5), including the column-eligibility pre-check (§2) and the loop-ordering optimization (§5.1).

This is the largest remaining implementation task in Phase 1 — larger than any of the six corruption runners individually, since it's the first piece of code in this project that trains and evaluates actual predictive models rather than manipulating data.
